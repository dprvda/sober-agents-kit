# Context framework

## What this is

A style guide for AI-authored docs in this repo. **Code is the source of truth for HOW** — AI reads code. **Docs are the source of truth for WHY, CURRENT-STATE, and DECISIONS** — what an AI cannot derive from reading the final-state code. The freshness gate (`check_doc_freshness.py`) stays as-is; this framework only changes what content goes inside the docs the gate already enforces.

## The four tiers

### T1 — Per-file `# REASON:` / `// REASON:` header (existing)

Every `.py` / `.sh` carries a `# REASON:` (or `// REASON:`) header at the top. ≥30 chars, substantive token. Enforced by `check_file_reason.py`. **Scope:** why this file exists; what bug or design intent it owns. Not enforced by THIS framework — already part of the existing gate roster.

### T2 — Per-module / per-folder `README.md`

Any folder that owns enough context to need a summary.

**Sections (in this order):**

- **Purpose** — 1 paragraph (≤120 words). Why this module exists as a separate unit from its peers. What problem it solves that wouldn't be solved by inlining its contents elsewhere.
- **Current state** — three sub-bullets: **Solid** (locked, tested), **Workaround** (known-temp fix + retire-when condition + citation), **Watch** (recently-touched / known-fragile).
- **Key decisions** — 3-7 bullets, each linking to an ADR (`[ADR-NNN]`) or an issue/commit citation. No vibe-bullets.
- **Gotchas** — ≥2 surprises a future AI would trip on if not warned. Each cited (`[#NNN]` / `[lessons_X]` / `[audit X §Y]`).
- **See also** — cross-links to related module READMEs, topic docs, ADRs.

**Length budget:** ≤200 lines (the `check_md_size` `per-module-readme` tier enforces a 9500-character cap).

**Anti-patterns** (do NOT include):

- Module-map tables (one row per source file). `git ls-files <module>/` is the module map.
- Test-surface catalogs ("22 inline tests + 4 integration"). The test runner list is the catalog.
- API rephrasing ("this fn takes X and returns Y"). The code IS the API.
- "Recent activity" sections. `git log -- <module>/` is the timeline.

### T3 — Cross-cutting topic doc (`docs/<topic>.md`)

Topics spanning ≥2 modules OR explaining a domain concept that doesn't have a natural code home. Same skeleton as T2 (Purpose / Current state / Key decisions / Gotchas / See also) plus any topic-specific structure already present.

**Length budget:** every top-level `docs/*.md` is the `check_md_size` `essential` tier — a 19500-character cap.

### T4 — Architecture Decision Record (`docs/decisions/ADR-NNN-slug.md`)

One ADR per locked design decision. Each ~30-80 lines following `docs/decisions/_TEMPLATE.md`:

- **Context** — the problem the project faced before the decision (≤200 words).
- **Considered** — bulleted alternatives. ≥2 rejected alternatives required. If you can't find ≥2, see "Honest-output principle" below.
- **Decision** — what's now locked (≤100 words). Reads as an instruction to future-self.
- **Why** — the rationale, cited (≥1 incident, commit, or empirical observation).
- **Status** — `locked` / `workaround` / `under-review` / `superseded-by ADR-NNN`.
- **Triggers for revisit** — ≤5 bullets of conditions that would make this decision worth re-examining.

ADRs are referenced from T2/T3 docs via `[ADR-NNN]` cross-links instead of inlining the WHY.

## The cascade workflow (when freshness fires)

The gate has TWO active checks:

- **Pass C `tracks-dir` (session-bounded)** — fires when BOTH conditions hold: (a) a file in the .md's declared `tracks_dir:` is newer than the .md AND (b) a new `.claude/handoffs/handoff_*.md` has landed since the .md's last update. Mid-session code iteration does NOT fire the cascade (no new handoff yet); the cascade only surfaces at session-wrap when the AI has fresh context to reconcile docs against the code that drifted. Body must update at that point (or `--ack-no-drift` with audit-logged reason).
- **Pass D `orphan-md`** — every authored `.md` must declare ONE of `tracks_dir:` (cascade-tracked living doc) OR `frozen_at: YYYY-MM-DD` (snapshot) OR `derived_from:` (presentation page, regenerated by a build step).

When Pass C fires, resolution:

1. The fire means it's session-wrap time. Re-read the tracking doc fully (lean — should be ≤200 lines for T2, ≤6000 tokens for T3).
2. Did the tracked code that changed touch context the doc claims to own?
   - **Yes — a decision changed, a workaround retired, a state shifted, a gotcha was discovered** → body-edit the relevant section. Cite the commit/issue.
   - **No — pure refactor, rename, formatting, internal restructure** → `--ack-no-drift PATH --reason "<≥30 chars why context didn't change>"`.
3. Many drifted docs at once? **Launch parallel agents** to reconcile in one pass — that's the explicit goal of the session-bounded design.
4. Commit.

**Always-context-changing patterns** (must body-edit):
- Changed `const` value where the doc cites the value.
- Added/removed public function/class/interface where the doc lists APIs.
- Changed error variants where docs reference them.
- A workaround moved from "Workaround" to "Solid" (or vice versa).
- An incident occurred that justified a new ADR.

**Usually implementation-only** (ack-OK):
- Whitespace / rename only.
- Internal function refactor (private fn body change).
- Test additions that don't change public behaviour.
- Comment-only edits.

## Citation style

Every WHY claim in T2/T3/T4 cites a source:

- Issue: `[#529]`
- Commit: `[abc1234]` (7-char SHA)
- Memory: `[lessons_some_topic]` / `[project_some_plan]`
- Audit: `[audit 25 §3]`
- Doc: `[some-topic.md §4]`
- ADR: `[ADR-001]`

Unsourced claims are the bug. If a claim has no source, it shouldn't be in the doc.

## The `tracks_dir:` convention

Prefer **file-level paths** over directory paths.

```yaml
# Preferred — file-level (fewer false cascades)
tracks_dir:
  - src/auth.py
  - src/signer.py

# Avoid — directory-level (fires on every file change anywhere under the dir)
tracks_dir:
  - src/
```

`check_doc_freshness.py` handles both shapes — its content-extension filter (`.py / .sh / .ts / .js / .md`) accepts single-file roots. File-level scopes mean fewer ripple-cascades when an unrelated file in the same module is touched.

Use directory paths only when every file in the directory affects the doc's claims.

## Honest-output principle

If you're writing an ADR or a "Key decisions" bullet and can't find ≥2 rejected alternatives + ≥1 incident or commit motivating the decision: **the decision was never really locked; it was an assumption that survived**. Write `INSUFFICIENT EVIDENCE — needs operator review` instead of fabricating context.

Quality > coverage. A 12-ADR set with real evidence beats a 15-ADR set with 3 vibe-ADRs.

## What NOT to do

- Don't write module-maps in READMEs.
- Don't write test catalogs.
- Don't rephrase code.
- Don't write historical timelines.
- Don't auto-pause: write a real-evidence ADR or write `INSUFFICIENT EVIDENCE`; never a vibe-ADR with hand-wavy "we picked this because it seemed right".
- Don't `--no-verify`. The freshness gate exists for a reason; if it fires, the right answer is body-edit OR substantive `--ack-no-drift --reason`.
